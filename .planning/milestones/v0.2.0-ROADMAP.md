# Milestone v0.2.0: Network

**Status:** ✅ SHIPPED 2026-05-26
**Phases:** 6-8
**Total Plans:** 12

## Overview

Advanced networking capabilities for ecs-sd: proxy mode for Fargate support, horizontal clustering for high availability, and internal Prometheus metrics for observability.

## Phases

### Phase 6: Proxy Mode & Fargate

**Goal:** Enable ecs-sd to act as a reverse proxy for discovered targets, unlocking Fargate support in segmented networks

**Depends on:** Phase 5 (v0.1.0 complete)

**Plans:** 3 plans

Plans:
- [x] 06-01: Config & AppState extensions (Mode enum, --public-address, ProxyTarget model)
- [x] 06-02: Fargate discovery (ENI IP extraction, Fargate branch in discover_all_clusters)
- [x] 06-03: Proxy handler + routes + /sd proxy-mode response + routing table rebuild

**Details:**

**Mode:** `ECS_SD_MODE=discovery` (default, v0.1.0 behavior) | `ECS_SD_MODE=proxy` (this phase)

**Requirements:**
- PROX-01: `--mode discovery|proxy` flag with `ECS_SD_MODE` env var
- PROX-02: `--public-address` / `ECS_SD_PUBLIC_ADDRESS` — ecs-sd's own reachable address for proxy-mode `/sd` targets
- PROX-03: In-memory routing table `HashMap<RouteId, ProxyTarget>` derived from discovery cache
- PROX-04: `/proxy/:id/*path` Axum handler — forward HTTP request to original target, stream response
- PROX-05: `/sd` in proxy mode returns `targets: ["<public-address>"]` with `__metrics_path__: "/proxy/<id>/metrics"`
- PROX-06: Routing table rebuilds on each cache refresh, same lifecycle as discovery
- PROX-07: Self-exclusion — ecs-sd's own `/metrics` endpoint excluded from proxy routing table
- FARG-01: Fargate task discovery — detect `launchType == FARGATE`, extract private IP from task ENI attachment
- FARG-02: Fargate targets included in routing table (proxy mode only)
- FARG-03: `--metadata-level` labels work identically for Fargate tasks

### Phase 7: Horizontal Clustering

**Goal:** Allow multiple ecs-sd instances to form a cluster with shared discovery state and single-leader AWS polling

**Depends on:** Phase 6 (routing table structure needed for gossip propagation in proxy mode)

**Plans:** 6 plans

Plans:
- [x] 07-01: Cargo.toml deps + cluster config fields (CLUS-01 to CLUS-04)
- [x] 07-02: Cluster module with ClusterState, is_leader, gossip publish/read (CLUS-05, CLUS-09)
- [x] 07-03: AppState extension + main.rs cluster wiring + follower sync (CLUS-06 to CLUS-08, CLUS-10)
- [x] 07-04: Integration tests: two-node cluster, cache propagation, failover, routing gossip
- [x] 07-05: README updates: cluster architecture, config reference, Docker Compose example
- [x] 07-06: Terraform module for Fargate deployment + operational runbook

**Details:**

**Clustering model:** Gossip-based membership (`chitchat` crate) + deterministic leader election (min NodeId among live members). Leader runs AWS discovery loop and gossips results; followers maintain local cache copy from gossip state. Any node serves HTTP.

**Requirements:**
- CLUS-01: `--cluster-mode standalone|cluster` flag with `ECS_SD_CLUSTER_MODE` env var
- CLUS-02: `--cluster-seeds` / `ECS_SD_CLUSTER_SEEDS` — comma-separated seed addresses
- CLUS-03: `--gossip-port` / `ECS_SD_GOSSIP_PORT` — gossip listener port (default: 8081)
- CLUS-04: `--node-id` / `ECS_SD_NODE_ID` — stable node identity
- CLUS-05: Leader election — node with lexicographically lowest NodeId among live members is leader
- CLUS-06: Leader runs discovery loop; followers skip discovery, consume gossiped cache state
- CLUS-07: Leader failure — gossip failure detection promotes next node within one gossip round
- CLUS-08: Cluster starts empty → elects leader → leader runs initial discovery → gossips results
- CLUS-09: Routing table (proxy mode) propagated via gossip alongside discovery cache
- CLUS-10: Standalone mode (`ECS_SD_CLUSTER_MODE=standalone`) — no gossip, identical to v0.1.0 behavior

### Phase 8: Internal Metrics & Self-Registration

**Goal:** Expose ecs-sd's own operational metrics at `/metrics` in Prometheus format, covering all subsystems (discovery, proxy, cluster)

**Depends on:** Phases 6 and 7 (proxy + cluster metrics require those subsystems to exist)

**Plans:** 3 plans

Plans:
- [x] 08-01: Core metrics infrastructure: prometheus crate, MetricsState, /metrics endpoint (MET-01, MET-02)
- [x] 08-02: Discovery & cache instrumentation: refresh loop metrics, integration tests (MET-02, MET-03)
- [x] 08-03: Conditional metrics & config: proxy metrics, cluster metrics, optional metrics port, self-registration docs (MET-04, MET-05, MET-06, MET-07)

**Details:**

**Requirements:**
- MET-01: `/metrics` endpoint returning Prometheus text exposition format
- MET-02: Discovery metrics — `ecs_sd_discovery_duration_seconds`, `ecs_sd_discovery_targets_total`, `ecs_sd_discovery_errors_total`
- MET-03: Cache metrics — `ecs_sd_cache_age_seconds`, `ecs_sd_cache_refreshes_total`
- MET-04: Proxy metrics — `ecs_sd_proxy_requests_total`, `ecs_sd_proxy_duration_seconds`
- MET-05: Cluster metrics — `ecs_sd_cluster_nodes_total`, `ecs_sd_cluster_is_leader`
- MET-06: `--metrics-port` / `ECS_SD_METRICS_PORT` — optional separate port for `/metrics`
- MET-07: Self-registration — ecs-sd with correct docker labels appears in its own `/sd` output

## Milestone Summary

**Key Decisions:**

- Use `chitchat` crate for gossip-based cluster membership (widely adopted, tokio-native)
- Deterministic leader election via min NodeId (simple, no additional consensus needed)
- Prometheus text format for metrics (industry standard, no schema negotiation)
- Optional separate metrics port for security separation
- Self-registration as emergent behavior (no special code, just standard discovery)

**Issues Resolved:**

- Fargate network segmentation — proxy mode bridges the gap
- Single point of failure — clustering enables HA deployment
- Operational blind spots — metrics expose internal state

**Issues Deferred:**

- Persistent cluster state (not needed for discovery cache)
- TLS for gossip (use VPC/network-level security)
- Custom metrics labels (future enhancement)

**Technical Debt:**

- None identified for this milestone
